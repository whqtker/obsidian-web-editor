# 코드베이스 리팩터링 전문가

전체 코드베이스(`src/`)를 대상으로 코드 간결성과 타입 안전성에 중점을 둔 리팩터링을 수행한다.

## 리팩터링 프로세스

### 1단계: 전체 스캔 및 이슈 식별

`src/` 디렉토리 전체를 스캔하여 아래 기준에 따른 이슈를 찾아라:

**타입 안전성**
- `any` 타입 사용
- 불필요한 타입 단언(`as`)
- 타입 가드 없는 `unknown` 처리
- API 응답의 타입 미검증

**코드 간결성**
- 중복 코드 (2곳 이상에서 동일 로직)
- 불필요한 변수/함수/import
- 과도한 중첩 (조기 반환으로 개선 가능)
- 불필요한 래퍼/추상화

**컨벤션 준수**
- Store 패턴: State와 Actions 인터페이스 분리, `create<T>()(...)` 커리 패턴
- 컴포넌트 패턴: 함수 컴포넌트 + named export, Props 인터페이스는 파일 내 정의
- Store 소비: 단일 값은 selector `(s) => s.value`, 여러 값은 destructure
- 이벤트 핸들러: 자식에 전달 시 useCallback 사용
- 에러 처리: API catch에서 `rethrowWithAuthCheck(err)` 필수, `err instanceof Error`로 안전 추출

### 2단계: 이슈 리포트

발견된 이슈를 아래 형식으로 우선순위별 정리하여 사용자에게 제시:

```
## 리팩터링 이슈 리포트

### 🔴 Critical (타입 안전성 위반)
- [ ] 파일:라인 — 설명

### 🟡 Important (코드 간결성)
- [ ] 파일:라인 — 설명

### 🟢 Minor (컨벤션 개선)
- [ ] 파일:라인 — 설명
```

사용자의 승인을 받은 후 다음 단계로 진행한다.

### 3단계: 수정 적용

승인된 이슈를 순차적으로 수정한다. 각 수정마다:
1. Before/After를 간략히 설명
2. 코드 변경 적용
3. 관련 import/export 업데이트

### 4단계: 검증

모든 수정 완료 후:
1. `npm run lint` 실행 — lint 에러 없음을 확인
2. `npm run test:run` 실행 — 테스트 통과를 확인
3. 실패 시 원인 분석 후 수정

## 주의사항

- 기능 변경 금지 — 동작은 동일하게 유지하면서 코드 품질만 개선
- 과도한 추상화 금지 — 현재 필요한 최소한의 변경만 수행
- `.obsidian/` 관련 코드의 보호 로직은 절대 제거하지 않음
- 테스트 파일은 소스 변경에 맞춰 최소한으로만 수정
